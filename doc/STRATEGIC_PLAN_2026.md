# omiai-server 战略级开发方案与演进计划 (2026-2027)

## 1. 战略目标回顾
基于“重度赋能B端（红娘/管理员）的婚恋CRM与业务流转系统”的核心定位，逐步向**“B2C闭环生态”**、**“AI深度赋能”**、**“数据资产挖掘”**和**“多租户SaaS生态”**演进，打造更高的商业壁垒。

结合当前团队的实际情况（5人以上全职研发团队，并行开发能力强，且对 AI 深度赋能有全方位诉求），本计划将通过高效的并行敏捷开发（Scrum），加速 AI 与 B 端业务闭环的融合。

## 2. 研发资源配置与阵型 (5+人全职团队)
建议将团队分为两个 Squad（敏捷小分队）并行推进：
- **业务基建 Squad (3人)**
  - 1名 后端架构师 (Go/DDD/SaaS改造主导)
  - 1名 前端研发 (Vue Admin + UniApp)
  - 1名 测试/QA (自动化测试保障)
  - **职责**：负责 Phase 1 的 B 端效能提升、风控中台、以及后续的 SaaS 改造。
- **AI 与算法 Squad (2人)**
  - 1名 AI/后端研发 (Go/Python，负责 LLM 对接、Prompt 工程、RAG 检索增强)
  - 1名 数据/算法工程师 (推荐系统构建、BI 漏斗分析)
  - **职责**：专注攻克三大 AI 核心场景（聊天摘要、智能推荐、虚拟红娘客服）。

## 3. 三阶段开发计划与排期 (Roadmap)

### Phase 1: 守正与基建 —— B端效能提升与 AI 引擎预研 (第1-1.5个月)
**目标**：巩固现有 `omiai-server` (V2) 架构，完善自动化审核流，同时启动 AI 基建。
**并行策略**：业务组主攻 B 端闭环，AI 组主攻模型选型与基建。

| 模块 | 核心功能点 | 技术方案评估与实施细节 | 负责人 / 预计工期 |
|---|---|---|---|
| **自助式风控与认证中台** | 接入第三方活体检测、实名认证、资产验证 | 调研并接入阿里云/腾讯云相关 API；在 `internal/service/auth_verify` 下新增模块。利用 GORM 完善认证状态机。 | 业务组 / 2 周 |
| **红娘效能 BI 看板** | 漏斗转化分析、红娘个人 KPI 追踪看板 | 扩展现有的 `dashboard` 模块；利用 Redis 缓存聚合报表数据，结合定时任务 (Cron) 夜间跑批计算。 | 业务组 / 2 周 |
| **数据合规与安全改造** | 敏感字段加密落盘、零信任访问审计日志 | 针对 GORM 模型引入自定义数据加密类型；在 `middleware` 层增加 `AuditLog` 审计中间件，记录关键查询。 | 业务组 / 1 周 |
| **AI 基础设施搭建** | LLM 统一网关、向量数据库选型 | 封装统一的 LLM 接口调用（如 OpenAI/DeepSeek API）；部署并对接向量数据库（Milvus 或 Qdrant）为后续推荐和知识库做准备。 | AI 组 / 2 周 |
| **AI 聊天摘要与打标签 (MVP)** | 企微聊天记录提取、自动生成画像标签 | 业务侧提供脱敏语料；设计 Prompt 提取结构化 JSON 标签，并自动更新至 `Client` 档案库。 | AI 组 / 3 周 |

### Phase 2: 出圈与智变 —— C端生态打通与全面 AI 赋能 (第1.5-3个月)
**目标**：孵化 `omiai-miniapp` C 端小程序，形成双向反馈；三大 AI 核心场景全面落地。

| 模块 | 核心功能点 | 技术方案评估与实施细节 | 负责人 / 预计工期 |
|---|---|---|---|
| **C端微信小程序 (v1.0)** | 客户登录、脱敏档案查看、候选人卡片、服务反馈 | 新增 C 端 API 路由组 (如 `/api/v1/c/`)；基于 JWT/Wechat Auth 的 C 端鉴权机制；打通 B 端跟进状态机。 | 业务组 / 4 周 |
| **消息与事件驱动中台** | 系统级消息通知、短信提醒解耦 | 引入 Kafka/RabbitMQ（当前基于 Redis queue），规范化 `internal/queues/` 消费者逻辑，实现异步解耦。 | 业务组 / 2 周 |
| **AI 智能推荐池 (Candidates)**| 基于用户深层偏好与行为的推荐算法 | 结合基础规则过滤 + 向量检索（根据 AI 提取的隐性标签计算余弦相似度）；重构 `ai_match.go`，提供智能算分接口。 | AI 组 / 3 周 |
| **AI 虚拟红娘客服** | 夜间/繁忙时自动接待新客户，完成初步资料收集 | 结合 RAG（检索增强生成）和公司话术库，打造客服 Agent；聊完后自动生成完整的 `Client` 档案分配给真人。 | AI 组 / 3 周 |

### Phase 3: 造血与生态 —— SaaS 化改造与上下游拓展 (第3-5个月)
**目标**：支持多租户（门店/独立红娘），为商业变现模式提供底层技术支撑。

| 模块 | 核心功能点 | 技术方案评估与实施细节 | 负责人 / 预计工期 |
|---|---|---|---|
| **多租户架构改造 (SaaS)** | 核心业务表增加 `tenant_id`；数据隔离；租户管理后台 | 这是重构深水区。在 GORM 注册全局 Scope 自动追加 `tenant_id` 过滤条件；重构 Context 传递租户信息。 | 全员 / 5 周 |
| **线下相亲场景预订** | 场地/餐厅预订对接、日程日历管家 | 建立资源池（Venue）领域模型；开发日历组件 API，防冲突调度算法。 | 业务组 / 3 周 |
| **动态定价模型与增值服务** | 自动评估脱单难度、提供报价建议、增值商城 | 基于历史成功匹配数据的统计分析模型；引入支付模块 (Wechat Pay / Alipay) 和商城基础订单系统。 | AI+业务组 / 3 周 |

## 4. 关键技术演进路线 (CTO 视角)

1. **领域驱动设计 (DDD) 的逐步落地**：
   目前的 `controller -> biz -> data` 架构在应对 SaaS 化和复杂 AI 逻辑时容易臃肿。建议在 Phase 2 期间，按**领域**（如：认证领域、匹配推荐领域、AI 客服领域、SaaS租户领域）进行水平拆分，边界清晰。

2. **AI 与核心业务的松耦合**：
   三大 AI 场景（标签摘要、推荐、虚拟客服）属于高耗时、重计算的操作。必须通过 **消息队列（Event-Driven）** 与主业务流程解耦。例如，微信聊天记录上传后，业务端立刻返回成功，后台异步调用 LLM 提取标签并存入向量库。

3. **数据库迁移管理 (Flyway 规范)**：
   继续严格遵守现有的 Flyway SQL 脚本迁移规范（如 `doc/sql/` 下的脚本）。Phase 3 的多租户改造将涉及大量的表结构变更，必须在测试环境经过完整的回归测试。

4. **测试与可观测性保障 (Observability)**：
   随着架构变复杂，尽早引入 Prometheus + Grafana 以及链路追踪（如 Jaeger/Skywalking），以便在 SaaS 化之后，能够快速定位慢查询和接口性能瓶颈。特别是 AI 接口调用的成功率、延迟和 Token 消耗，需要独立的大盘监控。

## 5. 里程碑检查点 (Milestones)
- **M1 (第4周)**：完成自助认证中台上线，红娘 BI 看板交付，AI 聊天标签提取跑通 MVP（脱敏数据测试）。
- **M2 (第8周)**：C 端小程序 V1.0 发布（核心信息展示与反馈），AI 智能推荐池初步替代人工低效筛选。
- **M3 (第12周)**：AI 虚拟客服上线接客，全面筹备 SaaS 数据库的平滑迁移。
- **M4 (第20周)**：完成 SaaS 多租户改造，系统支持向第三方加盟店或独立红娘开放售卖。
